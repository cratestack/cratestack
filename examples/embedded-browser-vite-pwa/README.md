# embedded-browser-vite-pwa

Installable [Progressive Web App](https://web.dev/progressive-web-apps/) built on the same Rust crate as [`embedded-browser-vite`](../embedded-browser-vite), with [`vite-plugin-pwa`](https://vite-pwa-org.netlify.app/) generating a Workbox service worker that precaches the app shell.

## What's different from `embedded-browser-vite`

| Concern | `embedded-browser-vite` | `embedded-browser-vite-pwa` |
|---|---|---|
| Data layer | OPFS-backed wasm SQLite | identical |
| App shell offline | only after first load while tab is open | **precached SW caches the page, JS, wasm** ‚Äî installable, offline-resilient |
| `manifest.webmanifest` | not generated | generated by `vite-plugin-pwa` |
| Service worker | none | generated (`dist/sw.js` + `workbox-*.js`) |
| Add-to-Home-Screen / installable | no | yes (subject to browser heuristics) |

The Rust source (`src/lib.rs`, `schema.cstack`, wasm-bindgen exports) is identical to the non-PWA variant ‚Äî only the JS-side `vite.config.ts` plus a small SW registration call in `main.ts` change.

## Run

```bash
cd examples/embedded-browser-vite-pwa/web
pnpm install
pnpm run dev
# Service worker registers in dev too (devOptions.enabled = true)
```

Production build:

```bash
pnpm run build
# Emits:
#   dist/sw.js                    ‚Äî generated service worker
#   dist/workbox-*.js             ‚Äî Workbox runtime
#   dist/manifest.webmanifest     ‚Äî installable web app manifest
#   dist/index.html               ‚Äî app shell, references the above
#   dist/assets/*.wasm            ‚Äî precached by the SW
```

Serve `dist/` over HTTPS (or `localhost`) to test installation:

```bash
pnpm run preview
```

## How the precache works

`vite-plugin-pwa` runs Workbox at build time. Its config (in `vite.config.ts`) globs `**/*.{js,css,html,wasm,svg}` from the build output and adds each file to the SW's precache manifest, with content-hashed URLs. The `maximumFileSizeToCacheInBytes: 3 * 1024 * 1024` line raises Workbox's per-asset budget so the ~960 KiB wasm chunk is allowed in.

Registration happens via `registerSW()` from `virtual:pwa-register` in `src/main.ts`. After registration:

- The SW starts serving precached files cache-first.
- Network failures don't break the app shell ‚Äî refreshing the tab while offline still loads everything.
- New SW versions notify the page (`onNeedRefresh`); the demo UI surfaces this in the status bar.

OPFS itself was already offline ‚Äî the database lives on the user's disk via `SyncAccessHandle`. The PWA wrapper extends offline-resilience to **app delivery**: when the user opens the page while disconnected, the cached SW serves the shell and the app boots normally.

## When to use this layout

- You want the app to be installable to a phone/desktop home screen with a real icon and manifest.
- The user might open the tab while offline (commuter trains, planes, flaky cafe Wi-Fi). The app should boot from the SW cache instead of failing to load.
- You're already shipping a Vite-based static site and want to add PWA features without rewriting your build chain.

If you don't need installability or precaching of the app shell ‚Äî the plain [`embedded-browser-vite`](../embedded-browser-vite) example is simpler and ships fewer assets.
