# embedded-expo

CrateStack's embedded shape on React Native, via [Expo](https://expo.dev) and a local native module that wraps a Rust cdylib. Same `include_embedded_schema!` + `ModelDelegate` + JSON FFI envelope as every other embedded example — what changes is the host: instead of wasm in a browser or `flutter_rust_bridge` bindings, the React Native side talks to the cdylib through `expo-modules-core`.

## Layout

```
embedded-expo/
├── README.md
├── native/                              # Rust cdylib (workspace member)
│   ├── Cargo.toml                       # crate-type cdylib + rlib + staticlib
│   ├── notes.cstack
│   └── src/lib.rs                       # cratestack_init / cratestack_dispatch / cratestack_free
├── modules/cratestack-notes/            # local Expo native module
│   ├── package.json                     # the `cratestack-notes` JS package (file: dep from app/)
│   ├── tsconfig.json
│   └── src/index.ts                     # typed accessors over the JSON envelope
│       └── (ios/ + android/             # GENERATED by `npx create-expo-module`)
└── app/                                 # Expo TypeScript app
    ├── package.json                     # depends on cratestack-notes via `file:../modules/...`
    ├── app.json
    ├── tsconfig.json
    └── App.tsx                          # the UI — list / add / mark done / delete
```

The Rust dispatcher in `native/src/lib.rs` is testable in isolation (`cargo test -p embedded-expo-native`). The Expo side is hand-written TypeScript; the iOS + Android native code that links the cdylib into the module is **generated separately** because it's heavy platform-specific scaffolding that's better produced by Expo's own tools than checked in by hand.

## Prerequisites

- Rust + cargo. For real iOS/Android builds:
  - Android: `cargo-ndk` (`cargo install cargo-ndk`) + Android Studio + NDK
  - iOS / macOS: Xcode + Xcode CLI tools
- Node.js 20+ and pnpm.
- The Expo CLI tooling: `npx create-expo-module@latest` is invoked at bootstrap time; `npx expo` runs your dev builds.

## Bootstrap (one-time per checkout)

```bash
cd examples/embedded-expo

# 1) Generate the native (iOS + Android) sources for the local module.
#    `create-expo-module` scaffolds Swift + Kotlin + the modules.json
#    pieces. We point it at the existing `modules/cratestack-notes`
#    directory so it adds platform code alongside our hand-written
#    src/index.ts.
cd modules
npx create-expo-module@latest cratestack-notes --no-prompt --with-readme=false \
    --description "cratestack-notes Expo native module" --no-example
cd ..

# 2) Edit the generated ios/CratestackNotesModule.swift and
#    android/.../CratestackNotesModule.kt: expose two Functions —
#    `initDatabase(path: String)` and `dispatch(requestJson: String)` —
#    that call into the Rust C ABI (`cratestack_init`,
#    `cratestack_dispatch`, `cratestack_free`). See
#    docs.expo.dev/modules/native-module-tutorial for the Function /
#    AsyncFunction syntax. The bytes returned by `cratestack_dispatch`
#    are the response JSON; pass them back to JS as a String.

# 3) Build the cdylib for each target you want to test on.
#    iOS Simulator on Apple Silicon:
cargo build -p embedded-expo-native --target aarch64-apple-ios-sim --release
#    iOS device:
cargo build -p embedded-expo-native --target aarch64-apple-ios --release
#    Android (cargo-ndk does all four arches in one go):
cargo ndk --target arm64-v8a --target armeabi-v7a --target x86 --target x86_64 \
    -o modules/cratestack-notes/android/src/main/jniLibs build --release -p embedded-expo-native

# 4) Install JS deps and run.
cd app
pnpm install
npx expo prebuild              # generates ios/ + android/ scaffolds for the *app*
npx expo run:ios               # or run:android
```

After this, regular development is `npx expo run:ios` / `npx expo run:android` from `app/`. Changes to `native/src/lib.rs` need a `cargo build` + a fresh `expo run`; changes to `modules/cratestack-notes/src/index.ts` or `app/App.tsx` hot-reload through Metro.

## How the wire protocol works

Same `OperationRequest` / `OperationResponse` JSON envelope used everywhere else cratestack does FFI:

```
JS:    { model: "Note", kind: "create", payload: { title, body, pinned } }
        |
        v   JSON.stringify → String → native.dispatch
        |
Native: cratestack_dispatch(bytes, len, &mut out_ptr, &mut out_len)
        |
        v   bytes back ← String back to JS
        |
JS:    { status: "ok", data: { id, title, body, pinned, completed, createdAt, updatedAt } }
```

The TS layer in `modules/cratestack-notes/src/index.ts` wraps this in typed functions: `listNotes()`, `createNote(input)`, `updateNote(id, patch)`, `deleteNote(id)`, `findNote(id)`. They all just construct the envelope, call the native bridge, and `JSON.parse` the response.

## Tests

```bash
cargo test -p embedded-expo-native        # dispatcher round-trips, error envelopes
```

Three tests:

- `create_then_find_unique_through_dispatcher` — full create → find_unique cycle through the JSON envelope.
- `unknown_model_returns_error_envelope` — request for a model that's not exposed returns `status: "err", code: "unknown_model"`.
- `dispatch_without_init_yields_not_initialized` — calling `dispatch` before `init_database` returns `status: "err", code: "not_initialized"`.

JS-level testing on the device happens once you run `npx expo run:ios` — there's no shortcut for that without the Expo runtime present.

## Verification status

Built and launched on an Android emulator (Pixel_10_Pro, API 36) and confirmed:

- ✅ `cargo-ndk` cross-compiles for all 4 ABIs (`arm64-v8a`, `armeabi-v7a`, `x86_64`, `x86`); `.so` files end up under `app/modules/cratestack-notes/android/src/main/jniLibs/<abi>/`.
- ✅ Gradle picks up the `.so` via `:cratestack-notes:mergeDebugNativeLibs` and packages them into the APK.
- ✅ APK installs cleanly on the emulator.
- ✅ Expo's autolinker discovers the local module (`cratestack-notes (0.7.10)` in the configure phase).
- ✅ App launches, JS bundle (708 modules) bundles via Metro and is delivered to the device.
- ✅ React Native's `Running "main" with {...fabric: true}` confirms the app entered the React root.
- ✅ `System.loadLibrary("embedded_expo_native")` succeeds (no `UnsatisfiedLinkError` in logcat).
- ⚠ Full JS-driven CRUD round-trip not yet snapshot-verified — `expo-file-system@55` moved `documentDirectory` to `expo-file-system/legacy`, fixed in `App.tsx` import; the HMR delta + reload cycle on a heavy first-launch emulator surfaced enough latency that I didn't get a clean confirmation here. Should work cleanly on a warm-cached subsequent run.

iOS path: same Rust crate, same C ABI exported via Swift's `@_silgen_name` in `app/modules/cratestack-notes/ios/CratestackNotesModule.swift`. The podspec vendors `libembedded_expo_native.a` (built by `cargo build --target aarch64-apple-ios-sim`) and declares `SystemConfiguration.framework`. **iOS Simulator runtime install required** (Xcode → Settings → Components → iOS 26.x) before `expo run:ios` works; macOS host alone isn't sufficient.

The Rust dispatcher itself is fully `cargo test`-verified — `cargo test -p embedded-expo-native` exercises the three envelope round-trips.
