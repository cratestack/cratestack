# embedded-browser-vite

Runnable demo: a CrateStack-driven note app whose entire data layer is `cratestack-rusqlite` compiled to `wasm32-unknown-unknown`, loaded by Vite, and persisted across browser reloads via OPFS.

## What it shows

- **`include_embedded_schema!`** generating typed model + descriptor + delegate code from `schema.cstack`.
- The same Rust `ModelDelegate` API used in the native [`embedded-cli`](../embedded-cli) example, but compiled to wasm and called from JavaScript.
- A **Dedicated Worker** that hosts the wasm runtime — required by the OPFS `SyncAccessHandle` API spec — talking to the main thread over `postMessage`.
- OPFS-backed persistence via the [`opfs-sahpool` VFS](https://crates.io/crates/sqlite-wasm-vfs), with a transparent **in-memory fallback** when OPFS isn't available (so the demo still works during initial development).
- A typed TypeScript wire protocol shared between the worker and main thread, with no hand-written `any` types.
- `wasm-pack`-generated `.d.ts` bindings so the worker calls the Rust functions with full type checking.

The schema is intentionally minimal (one `Note` model) so the focus stays on the wasm/Vite integration rather than schema features.

## Prerequisites

- **Rust toolchain** with the `wasm32-unknown-unknown` target:
  ```bash
  rustup target add wasm32-unknown-unknown
  ```
- **`wasm-pack`** on `PATH`:
  ```bash
  cargo install wasm-pack
  ```
- **Node.js 18+ and pnpm** (npm/yarn work too if you adjust the scripts).
- **A wasm-capable clang** — `sqlite-wasm-rs` builds SQLite's C source to wasm via `cc-rs`, and Apple's stock Xcode clang doesn't ship the wasm32 backend.

  **macOS:**
  ```bash
  brew install llvm
  ```
  The bundled npm scripts (`pnpm run dev` / `pnpm run build`) auto-detect Homebrew's LLVM at `/opt/homebrew/opt/llvm/bin/clang` (Apple Silicon) or `/usr/local/opt/llvm/bin/clang` (Intel). Set `CRATESTACK_CC` to an absolute path to override.

  **Linux:**
  ```bash
  sudo apt-get install clang lld   # Clang 14+
  ```
  The default `clang` on most distros already has the wasm32 backend; no extra config needed.

## Run the demo

```bash
cd examples/embedded-browser-vite/web
pnpm install
pnpm run dev
# Opens at http://localhost:5173
```

The page loads, spawns a Dedicated Worker that boots the wasm module and installs OPFS, then enables the form. Add a few notes, refresh the page — they're still there.

For a production build:

```bash
pnpm run build
pnpm run preview
```

## Layout

```
embedded-browser-vite/
├── Cargo.toml                  # cdylib + rlib targets
├── schema.cstack               # the .cstack schema (one Note model)
├── src/lib.rs                  # include_embedded_schema! + wasm-bindgen exports
├── web/
│   ├── package.json            # scripts: dev, build, typecheck
│   ├── vite.config.ts          # COOP/COEP headers + worker ESM mode
│   ├── tsconfig.json
│   ├── index.html              # the demo UI
│   ├── src/
│   │   ├── main.ts             # main-thread UI; spawns worker
│   │   ├── worker.ts           # Dedicated Worker hosting the wasm runtime
│   │   └── protocol.ts         # typed postMessage wire format
│   └── pkg/                    # generated by wasm-pack (gitignored after CI)
└── README.md
```

The shared `examples/scripts/wasm-build.mjs` helper sets `CC` for `cc-rs` so the wasm build picks up the right clang regardless of how the npm script was invoked.

## How the pieces connect

```
┌─ main thread ──────────────────────────────┐    ┌─ Dedicated Worker ──────────────┐
│ main.ts                                    │    │ worker.ts                       │
│   new Worker(URL('./worker.ts'))           │───▶│   await install_opfs()          │
│   worker.postMessage({ kind: 'add', … })   │    │   open_db('notes.db')           │
│   worker.onmessage → { ok, payload }       │◀───│   add_note / list_notes / …    │
│   render <ul> in DOM                       │    │   ↓                             │
│                                            │    │  wasm-bindgen Rust exports      │
│                                            │    │   ↓                             │
│                                            │    │  ModelDelegate over             │
│                                            │    │  RusqliteRuntime over           │
│                                            │    │  sqlite-wasm-rs + OPFS SAH-pool │
└────────────────────────────────────────────┘    └─────────────────────────────────┘
```

## Tests

`cargo test -p embedded-browser-vite-example` exercises the same delegate paths against an **in-memory** SQLite database on the **native** target. That gives `cargo test --workspace` coverage without needing a browser or wasm-pack. The browser path (OPFS persistence, Worker RPC) is verified by manually loading `pnpm run dev` and inspecting behavior.
